## Getting Started with Contributions

### 1. Choose Your Area

Pick an area that interests you:

- **Core Protocol**: Rust development, cryptography, Solana programs
- **SDK Development**: TypeScript, developer experience, APIs
- **Documentation**: Writing, examples, tutorials
- **Tooling**: CLI tools, testing frameworks, developer utilities
- **Examples**: Sample applications, demos, tutorials

### 2. Find Issues to Work On

Look for issues labeled with:

- `good first issue`: Perfect for newcomers
- `help wanted`: Community contributions welcome
- `documentation`: Documentation improvements needed
- `bug`: Bug fixes needed
- `enhancement`: New features or improvements

### 3. Set Up Your Development Environment

#### Core Protocol Development

```bash
# Clone the repository
git clone https://github.com/Lightprotocol/light-protocol.git
cd light-protocol

# Install Rust and Solana CLI
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
sh -c "$(curl -sSfL https://release.solana.com/stable/install)"

# Build the project
cargo build

# Run tests
cargo test
```

#### SDK Development

```bash
# Clone the repository
git clone https://github.com/Lightprotocol/zk-compression-sdk.git
cd zk-compression-sdk

# Install dependencies
npm install

# Build the SDK
npm run build

# Run tests
npm test
```

## Contribution Guidelines

### Code Standards

#### Rust Code (Core Protocol)

- Follow Rust standard formatting with `rustfmt`
- Use `clippy` for linting
- Write comprehensive tests
- Document public APIs with doc comments
- Follow Solana program best practices

```rust
/// Creates a new compressed account in the specified Merkle tree
///
/// # Arguments
/// * `ctx` - The program context
/// * `data` - The account data to compress
/// * `tree` - The target Merkle tree
///
/// # Returns
/// * `Result<()>` - Success or error
pub fn create_compressed_account(
    ctx: Context<CreateCompressedAccount>,
    data: Vec<u8>,
    tree: Pubkey,
) -> Result<()> {
    // Implementation
}
```

#### TypeScript Code (SDK)

- Use TypeScript strict mode
- Follow ESLint configuration
- Write unit and integration tests
- Document public APIs with JSDoc
- Use consistent naming conventions

```typescript
/**
 * Creates a new compressed token mint
 *
 * @param connection - Solana RPC connection
 * @param payer - Transaction fee payer
 * @param mintAuthority - Authority that can mint tokens
 * @param decimals - Number of decimal places
 * @returns Promise resolving to the mint public key
 */
export async function createMint(
  connection: Connection,
  payer: Keypair,
  mintAuthority: PublicKey,
  decimals: number
): Promise<PublicKey> {
  // Implementation
}
```

### Testing Requirements

#### Unit Tests

All new code must include unit tests:

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_create_compressed_account() {
        // Test implementation
    }
}
```

```typescript
describe("createMint", () => {
  it("should create a new compressed mint", async () => {
    // Test implementation
  });
});
```

#### Integration Tests

For significant features, include integration tests:

```typescript
describe("Token Operations Integration", () => {
  it("should mint and transfer tokens end-to-end", async () => {
    // End-to-end test
  });
});
```

### Documentation Standards

#### Code Documentation

- Document all public APIs
- Include usage examples
- Explain complex algorithms
- Document error conditions

#### README Files

Each repository should have:

- Clear project description
- Installation instructions
- Usage examples
- Contribution guidelines
- License information

#### Changelog

Maintain a changelog following [Keep a Changelog](https://keepachangelog.com/):

```markdown
## [1.2.0] - 2024-01-15

### Added

- New batch transfer functionality
- Support for custom Merkle tree configurations

### Changed

- Improved error handling in SDK methods
- Updated dependencies to latest versions

### Fixed

- Fixed memory leak in proof generation
- Resolved race condition in concurrent operations
```

## Pull Request Process

### 1. Fork and Branch

```bash
# Fork the repository on GitHub
# Clone your fork
git clone https://github.com/yourusername/light-protocol.git

# Create a feature branch
git checkout -b feature/your-feature-name
```

### 2. Make Changes

- Write clean, well-documented code
- Add tests for new functionality
- Update documentation as needed
- Follow the existing code style

### 3. Test Your Changes

```bash
# Run all tests
cargo test  # For Rust projects
npm test    # For TypeScript projects

# Run linting
cargo clippy  # For Rust projects
npm run lint  # For TypeScript projects

# Check formatting
cargo fmt --check  # For Rust projects
npm run format     # For TypeScript projects
```

### 4. Submit Pull Request

- Write a clear PR title and description
- Reference related issues
- Include screenshots for UI changes
- Request review from maintainers

#### PR Template

```markdown
## Description

Brief description of the changes

## Type of Change

- [ ] Bug fix
- [ ] New feature
- [ ] Breaking change
- [ ] Documentation update

## Testing

- [ ] Unit tests pass
- [ ] Integration tests pass
- [ ] Manual testing completed

## Checklist

- [ ] Code follows style guidelines
- [ ] Self-review completed
- [ ] Documentation updated
- [ ] Tests added/updated
```

## Release Process

### Versioning

We follow [Semantic Versioning](https://semver.org/):

- **MAJOR**: Breaking changes
- **MINOR**: New features (backward compatible)
- **PATCH**: Bug fixes (backward compatible)

### Release Checklist

1. Update version numbers
2. Update changelog
3. Run full test suite
4. Create release PR
5. Tag release after merge
6. Publish packages
7. Update documentation
